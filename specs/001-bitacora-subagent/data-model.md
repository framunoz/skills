# Data Model — Logbook Subagent

All artifacts live in the **consumer project** under `logbook/<slug>/`. The subagent repo ships only code to read/write these artifacts.

## Directory layout (per logbook, inside a consumer project)

```text
<project-root>/
└── logbook/
    └── <slug>/                 # slug matches ^[a-z0-9]+(-[a-z0-9]+)*$
        ├── meta.json           # logbook metadata (declared once at init)
        ├── entries.jsonl       # append-only entry stream, one JSON object per line
        └── rendered.md         # optional, regenerated by `format.py` (derived)
```

Multiple logbooks coexist as siblings under `logbook/`. The `rendered.md` file is a derived artifact — never read by `push.py`, always overwritten by `format.py`.

## `meta.json`

```json
{
  "slug": "tests-login",
  "schema_type": "tests",
  "title": "Login flow tests",
  "description": "Results of manual and automated login tests.",
  "created_at": "2026-04-18T10:32:00Z",
  "format_version": 1
}
```

**Validation rules**:

- `slug` MUST match `^[a-z0-9]+(-[a-z0-9]+)*$` and MUST equal the parent directory name.
- `schema_type` ∈ `{"tests", "collaboration", "free"}`.
- `created_at` MUST be RFC 3339 / ISO 8601 UTC.
- `format_version` is an integer; bumped when the on-disk layout changes.
- Immutable after creation except `title` and `description` (which may be edited manually; the subagent does not edit them).

## `entries.jsonl`

One JSON object per line. Lines are appended chronologically by `push.py`. No line is ever modified or deleted.

### Common fields (present on every entry)

| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | integer | yes | Monotonic per-logbook counter starting at 1. |
| `ulid` | string | yes | ULID for stable cross-reference. |
| `created_at` | string (RFC 3339) | yes | UTC timestamp assigned by `push.py`. |
| `type` | string | yes | One of `tests`, `collaboration`, `free`, `amendment`. MUST match the logbook's `schema_type` except for `amendment`, which is allowed in any logbook. |
| `title` | string | yes | ≤ 200 chars. |
| `tags` | string[] | no | Lowercase, hyphenated; free-form. |
| `author` | string | no | Free-text ("user", "user+claude", "claude-assisted", etc.). Default `"user"`. |

### Type-specific fields

**`tests`** (for `schema_type: tests`):

| Field | Type | Required | Notes |
|---|---|---|---|
| `went_well` | string[] | no* | Each item a short sentence. |
| `went_wrong` | string[] | no* | Each item a short sentence. |
| `next_steps` | string[] | no | |
| `context` | string | no | Free-form narrative. |

\* At least one of `went_well` or `went_wrong` MUST be non-empty; both empty is rejected. Missing sections are rendered as "No observations" (FR-009); they are NOT fabricated.

**`collaboration`** (for `schema_type: collaboration`):

| Field | Type | Required | Notes |
|---|---|---|---|
| `ai_contribution` | string | no* | What the AI produced/proposed. |
| `human_contribution` | string | no* | What the human designed/decided. |
| `human_corrections` | string[] | no | Each item = one correction the human made to the AI. |
| `milestone` | string | no | Optional anchor for retrospectives. |
| `context` | string | no | |

\* At least one of `ai_contribution` or `human_contribution` MUST be non-empty.

**`free`** (for `schema_type: free`):

| Field | Type | Required | Notes |
|---|---|---|---|
| `body` | string | yes | Markdown permitted. |

**`amendment`** (allowed regardless of logbook `schema_type`):

| Field | Type | Required | Notes |
|---|---|---|---|
| `amends` | object `{id:int, ulid:string}` | yes | Both values MUST reference an existing entry in the same logbook; `push.py` verifies. |
| `reason` | string | yes | Short explanation: typo, fact correction, clarification, redaction. |
| `body` | string | yes | Corrected or clarifying content. |

### Example entries (one per line in `entries.jsonl`)

```json
{"id":1,"ulid":"01HW...A1","created_at":"2026-04-18T10:45:12Z","type":"tests","title":"Smoke test after login rework","went_well":["OAuth redirect works"],"went_wrong":["Refresh loop on stale cookie"],"next_steps":["Reproduce in staging"],"tags":["auth","smoke"],"author":"user"}
{"id":2,"ulid":"01HW...B2","created_at":"2026-04-18T11:02:04Z","type":"amendment","amends":{"id":1,"ulid":"01HW...A1"},"reason":"Clarification","body":"The refresh loop only affects Safari; Chrome and Firefox are fine.","title":"Amend #1: refresh loop scope","author":"user"}
```

## Entity summary

- **Logbook**: `logbook/<slug>/` directory; identified by `slug`; has exactly one `schema_type`; contains one `meta.json` and one `entries.jsonl`.
- **Entry**: one line in `entries.jsonl`; identified by `(logbook_slug, id)` and independently by `ulid`; immutable once written.
- **Amendment**: a specialized entry that references an earlier entry's `(id, ulid)` in the same logbook.

## Invariants

1. Every existing line in `entries.jsonl` is byte-identical across pushes (SC-003).
2. `id` values are strictly increasing and contiguous starting at 1; `push.py` reads the last line to compute the next `id`.
3. `ulid` values are globally unique (collision probability negligible).
4. An `amendment` whose `amends.id` or `amends.ulid` does not match any existing entry is rejected before writing.
5. `rendered.md` is always derived; losing it is not a data loss event.
